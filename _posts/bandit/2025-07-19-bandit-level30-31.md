## 1. 문제 개요

> **Level Goal**
> 
> There is a git repository at `ssh://bandit30-git@localhost/home/bandit30-git/repo`.
> The password for the next level is the same as for the user bandit30.

이번에도 저장소를 복제했지만, `git log`나 브랜치 확인(`git branch -a`)을 해도 비밀번호가 보이지 않았습니다. Git에는 커밋(Commit)과 브랜치(Branch) 외에도 **태그(Tag)** 라는 중요한 기능이 있다는 것을 알았습니다. 개발자가 "버전 1.0 출시"처럼 중요한 시점에 깃발을 꽂아두는 기능입니다.

---

## 2. 사용 명령어

| 명령어 | 설명 |
|--------|------|
| `git tag` | 저장소에 있는 모든 태그 목록을 보여줌 |
| `git show [태그명]` | 해당 태그가 가리키는 커밋의 내용과 태그 메시지를 보여줌 |

---

## 3. 풀이 과정

`bandit30` 계정으로 로그인합니다.

```bash
ssh bandit30@bandit.labs.overthewire.org -p 2220
```

### 1. Clone
`/tmp`에 저장소를 복제합니다.

```bash
mkdir /tmp/mygit30
cd /tmp/mygit30
git clone ssh://bandit30-git@localhost/home/bandit30-git/repo
cd repo
```

### 2. README 확인
그냥 빈 파일이거나 별 내용이 없습니다.

### 3. 태그 확인
`git tag` 명령어로 태그가 있는지 봅니다.

```bash
bandit30@bandit:/tmp/mygit30/repo$ git tag
secret
```
`secret`이라는 아주 수상한 이름의 태그가 있습니다.

### 4. 태그 내용 확인
`git show` 명령어로 이 태그가 무엇을 가리키는지 봅니다.

```bash
bandit30@bandit:/tmp/mygit30/repo$ git show secret
tag secret
Tagger: ...
Date: ...

47e603bb42...
```
출력 결과 중간에 비밀번호가 섞여 있을 것입니다.

```
47e603bb42...
```

## 5. 배운 점

1. **Git Tag**: 커밋 아이디(해시값)는 기억하기 어렵기 때문에 `v1.0`, `release-2023` 같이 사람이 읽기 쉬운 이름을 붙여두는 기능입니다.
2. **숨겨진 정보**: 해커는 저장소 내의 모든 객체(Blob, Tree, Commit, Tag)를 다 뒤져봅니다. 브랜치나 로그에 없다고 안심하면 안 됩니다.

---

## 6. 보안 관점

- **릴리스 관리**: 태그는 보통 배포 버전을 관리할 때 씁니다. 만약 실수로 비밀번호가 포함된 버전에 태그를 붙여 배포했다가 나중에 코드를 수정했더라도, 태그 자체를 삭제하지 않으면 그 시점의 코드는 여전히 조회 가능합니다.
