## 1. 문제 개요

> **Level Goal**
> 
> The password for the next level is stored in `/etc/bandit_pass/bandit14` and can only be read by user bandit14. For this level, you don't get the next password, but you get a **private SSH key** that can be used to log into the next level.

이번 레벨은 비밀번호 대신 **SSH 개인키**를 획득하는 문제였습니다. 이 키를 사용하여 비밀번호 없이 `bandit14` 계정으로 로그인해야 했습니다.

---

## 2. 사용 명령어

| 명령어 | 설명 |
|--------|------|
| `ssh -i [키파일]` | 지정한 **개인키(Identity File)**을 사용하여 인증 시도 |
| `localhost` | 네트워크상의 자기 자신(현재 서버)을 가리키는 호스트명 |

---

## 3. 풀이 과정

`bandit13` 계정으로 로그인합니다.

```bash
ssh bandit13@bandit.labs.overthewire.org -p 2220
```

### 1. SSH 키 확인
홈 디렉토리에 `sshkey.private` 파일이 있습니다. 이것이 다음 레벨(`bandit14`)로 갈 수 있는 열쇠(Private Key)입니다.
```bash
bandit13@bandit:~$ ls
sshkey.private
```

### 2. 키를 이용한 접속
보통은 내 컴퓨터에서 서버로 접속하지만, 이번엔 **서버 내부에서(localhost) 다른 사용자로 접속**하는 것입니다.
`-i` 옵션으로 키 파일을 지정해서 접속합니다.

```bash
bandit13@bandit:~$ ssh -i sshkey.private bandit14@localhost -p 2220
```
비밀번호를 묻지 않고 바로 접속됩니다!

### 3. 비밀번호 확인
`bandit14`로 로그인된 상태입니다. 비밀번호 파일을 읽어봅니다. 이제 문제에서 말한 비밀번호 파일(`/etc/bandit_pass/bandit14`)을 읽을 수 있습니다.

```bash
bandit14@bandit:~$ cat /etc/bandit_pass/bandit14
4YjL3GvafeR3b2hTUtbT3bTGo22a4D22
```

## 5. 배운 점

1. **SSH 키 기반 인증**: 비밀번호 대신 암호화 키 쌍(Public Key / Private Key)을 사용하여 더 안전하고 편리하게 로그인할 수 있습니다.
2. **Localhost**: 자기 자신을 가리키는 네트워크 주소(`127.0.0.1`)입니다. 서버 내부에서 다른 서비스를 테스트할 때 자주 사용합니다.

---

## 6. 보안 관점

- **개인키 유출의 위험성**: `sshkey.private` 같은 개인키 파일은 "만능열쇠"와 같습니다. 이 파일이 유출되면 해당 키가 등록된 모든 서버에 비밀번호 없이 접속할 수 있게 됩니다.
- 따라서 개인키 파일은 반드시 **타인이 읽을 수 없도록 권한 관리(`chmod 600`)** 를 철저히 해야 하며, 실수로 GitHub 같은 공용 저장소에 올리지 않도록 주의해야 합니다.
