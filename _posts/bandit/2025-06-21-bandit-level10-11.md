## 1. 문제 개요

> **Level Goal**
> 
> The password for the next level is stored in the file `data.txt`, which contains **base64 encoded data**.

비밀번호가 평문이 아닌 **Base64**라는 방식으로 인코딩되어 `data.txt`에 저장되어 있었습니다. 이를 디코딩(복호화)해서 원래의 비밀번호를 알아내야 했습니다.

---

## 2. 사용 명령어

| 명령어 | 설명 |
|--------|------|
| `base64` | 데이터를 Base64 형식으로 인코딩하거나 디코딩 |
| `-d` 또는 `--decode` | Base64로 인코딩된 데이터를 원래대로 복원(Decode) |

### Base64란?
바이너리 데이터(이미지, 실행 파일 등)를 텍스트(ASCII 문자) 형태로 변환하여 전송하거나 저장하기 위해 사용하는 인코딩 방식입니다. 암호화가 아니므로 누구나 쉽게 복원할 수 있습니다.

---

## 3. 풀이 과정

`bandit10` 계정으로 로그인합니다.

```bash
ssh bandit10@bandit.labs.overthewire.org -p 2220
```

### 1. 파일 확인 (Base64 판별)
`data.txt` 파일을 열어보니 알 수 없는 문자들이 가득합니다.
문자열 끝에 `==`가 붙어있는 것을 보니 **Base64**로 인코딩된 데이터임을 짐작할 수 있습니다.

```bash
bandit10@bandit:~$ cat data.txt
VGhlIHBhc3N3b3JkIGlzIDY... (알 수 없는 문자열)
```

일반적인 문장과는 다르게 알파벳 대소문자와 숫자, `+`, `/`, `=` 기호로 이루어진 문자열이 보입니다. 전형적인 Base64 데이터의 특징입니다.

### 2. 디코딩 (Decoding)
리눅스에는 `base64`라는 명령어가 기본으로 있습니다. `-d` (decode) 옵션을 사용하면 원래 내용으로 되돌릴 수 있습니다.

```bash
bandit10@bandit:~$ base64 -d data.txt
The password is 6TSfY9J3er2d2a45TQLNtuAon22a1T3a
```
(또는 `cat data.txt | base64 -d` 로 파이프를 사용할 수도 있습니다)

## 5. 배운 점

1. **인코딩 vs 암호화**: 인코딩은 데이터 형식을 바꾸는 것일 뿐, 비밀번호를 숨기는 보안 기능(암호화)이 아닙니다. Base64로 된 비밀번호는 누구나 풀 수 있습니다.
2. **Base64 식별**: 문자열 끝에 `==` 패딩이 있거나, `A-Z, a-z, 0-9, +, /` 문자로만 구성되어 있다면 Base64를 의심해볼 수 있습니다.

---

## 6. 보안 관점

- **취약한 데이터 저장**: 중요한 정보(비밀번호, 개인정보)를 Base64로만 인코딩해서 저장하거나 전송하는 것은 매우 위험합니다. ("Security through obscurity"의 나쁜 예)
- 공격자는 Base64 문자열을 보면 즉시 디코딩을 시도합니다. 반드시 적절한 암호화(Encryption)나 해싱(Hashing)을 사용해야 합니다.
