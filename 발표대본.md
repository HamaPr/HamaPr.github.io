**[Slide 23: 아키텍처 핵심 목표]**

데이터 계층을 설계하면서 제가 집중한 4가지 핵심 목표입니다.

셋째, 부하 분산을 통한 **성능 최적화**,
마지막으로, 데이터 수명 주기에 따른 **비용 효율성**입니다.

**[Slide 24: MySQL Flexible Server 상세]**

먼저 데이터베이스입니다. Azure MySQL Flexible Server를 도입하며 **Zone-Redundant(가용 영역 중복)** 구성을 적용했습니다.

주 서버와 대기 서버를 물리적으로 다른 존(Zone)에 배치하여, 주 서버 장애 시 데이터 유실(RPO) 없이 즉시 대기 서버로 **자동 장애 조치(Auto-Failover)**가 이루어집니다. 또한, 트래픽 폭주에 대비해 **2개의 Read Replica(읽기 복제본)**를 두어 읽기 요청을 분산시키는 CQRS 패턴으로 메인 서버의 부하를 획기적으로 낮췄습니다. 여기에 더해, 정형화되지 않은 유연한 데이터 처리를 위해 **Cosmos DB**를 보조 저장소로 채택하여 데이터 성격에 따른 이원화 전략을 취했습니다.

**[Slide 25: Storage & Data Management 전략]**

스토리지 전략에서는 **비용 효율성**에 집중했습니다.
테라폼을 이용해 **수명 주기 정책(Lifecycle Policy)**을 코드로 구현했습니다. 생성 후 30일, 90일이 지난 데이터는 자동으로 저렴한 Cool/Archive 티어로 이동하게 하여, 핫 티어 대비 최대 90% 이상의 비용 절감을 달성했습니다. 또한, 데이터 보존을 위해 **ZRS(Zone-Redundant Storage)** 옵션을 적용하여, 물리적 데이터센터 장애 시에도 데이터 내구성을 보장했습니다.

**[Slide 26: 성능 가속화 계층]**

성능을 위한 가속화 계층입니다.
DB 앞단에는 Redis Cache를 배치해 자주 쓰는 데이터를 메모리에서 바로 응답하게 했고, 정적 콘텐츠는 CDN(Front Door)을 통해 전 세계 어디서든 빠르게 로딩되도록 하여 대역폭 비용과 속도를 모두 잡았습니다.

**[Slide 27: 백업 자동화 시나리오]**

마지막으로 백업입니다.
운영 DB에 부하를 주지 않도록 Azure Data Factory를 활용한 서버리스 파이프라인을 구축했습니다. 매일 정해진 시간에 데이터를 안전하게 추출 및 아카이빙하여, 랜섬웨어 등의 위협으로부터 **특정 시점 복구(PITR)**가 가능하도록 구현했습니다. 또한, 이 모든 리소스에서 발생하는 로그는 **Log Analytics**로 수집됩니다. 장애 발생 시 언제, 어디서, 왜 문제가 생겼는지 즉각 추적할 수 있는 **중앙 집중형 관제 환경**을 마련했습니다.

### 1. 핵심 기술 용어 정리
#### **데이터베이스 (DB) 관련**
*   **Zone-Redundant (가용 영역 중복):** 하나의 리전(예: 서울) 안에 있는 서로 다른 데이터센터(Zone 1, Zone 2)에 서버를 나눠서 배치하는 것. 하나가 불타도 옆 건물에서 서비스가 돌아가는 원리.
*   **Auto-Failover (자동 장애 조치):** 주 서버(Master)가 죽으면, 사람이 개입하지 않아도 시스템이 알아서 대기 서버(Standby)를 주 서버로 승격시키는 기술. "자동 교체"라고 생각하면 됨.
*   **Synchronous Replication (동기식 복제):** 주 서버에 데이터가 써질 때, 대기 서버에도 동시에 써지는 방식. 속도는 약간 느려질 수 있지만 데이터가 100% 일치함(유실 없음).
*   **CQRS (Command Query Responsibility Segregation):** '명령(쓰기)'과 '조회(읽기)'를 분리하는 패턴. 쓰기는 Master DB에, 읽기는 Replica DB에 시켜서 일을 나눠주는 것.
*   **Read Replica (읽기 복제본):** Master DB의 데이터를 복사해서 가지고 있는 '조회 전용' DB.

#### **스토리지 & 기타 관련**
*   **ZRS (Zone-Redundant Storage):** 스토리지 데이터를 한 리전 내의 3개 가용 영역에 분산 저장하는 것. (LRS는 데이터센터 1개에만 저장, GRS는 아예 다른 지역(예: 부산)에 저장).
*   **Lifecycle Policy (수명 주기 정책):** 데이터의 나이에 따라 저장 위치를 자동으로 옮기는 규칙. (자주 쓰는 핫 → 가끔 쓰는 쿨 → 거의 안 쓰는 아카이브).
*   **ETL (Extract, Transform, Load):** 데이터를 추출(E)해서, 변환(T)하고, 저장(L)하는 과정. 여기서는 DB에서 데이터를 뽑아 스토리지로 옮기는 백업 과정을 뜻함.
*   **PITR (Point-In-Time Recovery):** "어제 오후 2시 30분 시점으로 돌려줘"처럼 특정 시간으로 데이터를 복구하는 기능.
*   **LRU (Least Recently Used):** Redis 캐시 메모리가 꽉 찼을 때, "가장 오랫동안 안 쓴 데이터"부터 갖다 버리는 알고리즘.

### 2. 예상 질문 & 답변
#### **Q1. (HA) Zone-Redundant 구성을 하면 비용이 비싸지 않나요?**
> **답변:** 네, 단일 구성보다 비용은 증가합니다. 하지만 저희 서비스는 데이터 무결성이 무엇보다 중요하다고 판단했습니다. 서버 장애로 인한 서비스 중단 시 발생하는 비즈니스 손실 비용이 인프라 비용보다 훨씬 크기 때문에, 과감하게 **고가용성 아키텍처(ZRS/Zone-Redundant)**에 투자했습니다.

#### **Q2. (성능) 동기식 복제(Synchronous Replication)를 하면 DB 쓰기 속도가 느려지지 않나요?**
> **답변:** 맞습니다. 대기 서버에 기록이 완료될 때까지 기다려야 하므로 약간의 지연(Latency)이 발생할 수 있습니다. 하지만 Azure의 내부 백본망 속도가 매우 빠르고, 데이터 유실(RPO)을 '0'으로 만드는 것이 속도보다 중요했기 때문에 감수할 만한 트레이드오프라고 판단했습니다.

#### **Q3. (백업) Azure MySQL 자체 백업 기능이 있는데 굳이 ADF(Data Factory)를 쓴 이유가 뭔가요?**
> **답변:** Azure 자체 백업은 편리하지만 최대 보존 기간 제한이 있고 복원 비용이 비쌀 수 있습니다. 저희는 컴플라이언스 준수를 위해 데이터를 **장기 보관(아카이빙)**해야 한다고 가정했고, 저렴한 Archive Tier 스토리지로 데이터를 옮기기 위해 ADF 파이프라인을 추가로 구성했습니다.

#### **Q4. (스토리지) 스토리지 비용 절감이 최대 90%라고 했는데, Archive Tier의 단점은 없나요?**
> **답변:** Archive Tier는 저장 비용은 싸지만, 데이터를 다시 꺼낼 때(Rehydrate) 시간과 비용이 많이 듭니다. 따라서 자주 접근하는 데이터가 아니라, 법적 근거로 보관만 해두는 **'콜드 데이터'** 위주로 정책을 적용하여 단점을 상쇄했습니다.

#### **Q5. (보안) Private Endpoint를 쓰면 외부에서 DB 관리는 어떻게 하나요?**
> **답변:** 공인 IP가 없기 때문에 외부 직접 접속은 불가능합니다. 대신 아키텍처에 구성된 **Bastion Host(점프 서버)**를 통해 내부망으로 진입한 뒤, 거기서 DB에 안전하게 접속하여 관리하고 있습니다.

#### **Q6. (캐시) Redis가 죽으면 서비스도 멈추나요?**
> **답변:** 아닙니다. Redis는 성능을 높여주는 '캐시' 역할일 뿐입니다. Redis가 다운되더라도 요청은 원본 DB(MySQL)로 전달되어 서비스는 계속 유지됩니다. 다만 DB 부하가 일시적으로 늘어날 수는 있습니다.

#### **Q7. (복제) Read Replica를 2개나 둔 기준은 무엇인가요?**
> **답변:** 웹 서비스 특성상 글을 쓰는(Write) 행위보다 글을 보는(Read) 행위가 압도적으로 많습니다. 일반적으로 8:2 정도의 비율을 가정했을 때, 주 서버 1대의 부하를 효과적으로 분산하기 위해 2대의 읽기 전용 서버를 배치했습니다. 추후 트래픽이 늘어나면 최대 10대까지 늘릴 수 있습니다.

#### **Q8. (Terraform) DB 설정을 콘솔이 아니라 Terraform으로 했을 때 어려웠던 점은?**
> **답변:** DB의 파라미터나 방화벽 규칙 등 세세한 설정값을 코드로 정의하는 것이 처음에는 복잡했습니다. 하지만 한 번 코드로 작성해 두니, 실수 없이 똑같은 환경을 재배포할 수 있어서 **인프라의 불변성(Immutability)**을 확보할 수 있었습니다.

#### **Q9. (네트워크) Private Endpoint와 Service Endpoint의 차이는 무엇이고 왜 Private를 썼나요?**
> **답변:** Service Endpoint는 여전히 공용 IP 대역을 사용하지만, Private Endpoint는 우리 VNet 내부의 **사설 IP**를 할당받습니다. 즉, DB를 우리 내부망에 있는 서버처럼 완벽하게 격리할 수 있어서 보안성이 훨씬 뛰어난 Private Endpoint를 선택했습니다.

#### **Q10. (아키텍처) GRS(지리적 중복)를 쓰지 않고 ZRS(영역 중복)만 쓴 이유는?**
> **답변:** 예산과 프로젝트 규모를 고려한 현실적인 타협안입니다. 한국 리전 전체가 파괴되는 재해 상황(Disaster)까지 대비하려면 GRS가 필요하지만, 학생 프로젝트 수준에서는 **데이터센터 단위의 장애(ZRS)**만 대비해도 충분히 높은 가용성을 확보할 수 있다고 판단했습니다.

### 📂 06. Storage (스토리지 및 데이터 관리)

**1. `06_01_storage_account.tf`**
*   **역할:** 메인 저장소(Blob Storage) 생성.
*   **핵심 기술:** **GRS/ZRS (Geo/Zone Redundant Storage)**
    *   데이터를 단순히 저장하는 게 아니라, 물리적으로 다른 위치(Zone/Region)에 실시간 복제하여 데이터센터 화재가 나도 데이터가 살아남게 설계함. **(내구성 확보)**

*   **역할:** 저장소 내부의 '폴더' 격인 컨테이너 생성 (media, backup, tfstate 등).
*   **핵심 기술:** **Access Level Control (접근 제어)**
    *   모든 컨테이너를 `Private`으로 설정하여, 익명 사용자가 URL만으로 데이터에 접근하는 것을 원천 차단함. **(보안 기초)**

**3. `06_03_lifecycle_policy.tf`**
*   **역할:** 오래된 데이터를 싼 저장소로 자동 이동시키는 규칙.
*   **핵심 기술:** **Tiering Automation (계층화 자동화)**
    *   운영자가 일일이 파일을 옮길 필요 없이, 설정된 기간(30일, 90일)에 따라 Hot → Cool → Archive로 이동시켜 스토리지 비용을 획기적으로 줄임. **(비용 최적화)**

**4. `06_04_cdn_profile.tf`**
*   **역할:** Azure Front Door(CDN) 서비스의 껍데기(프로필) 생성.
*   **핵심 기술:** **Global Edge Network**
    *   전 세계에 퍼진 MS의 엣지 네트워크를 활용할 준비를 하는 단계.

**5. `06_05_cdn_endpoint.tf`**
*   **역할:** 실제 CDN 접속 주소 생성 및 캐싱 규칙 설정.
*   **핵심 기술:** **Dynamic Caching & Compression**
    *   스토리지에 있는 이미지/영상을 엣지 서버에 저장(캐싱)하고, 압축 전송을 활성화하여 사용자 로딩 속도를 높임. **(성능 가속)**

**6. `06_06_private_endpoint.tf`**
*   **역할:** 스토리지를 내부망(VNet)에 연결.
*   **핵심 기술:** **Private Link (프라이빗 링크)**
    *   스토리지를 공용 인터넷망에서 숨기고, 오직 내부 네트워크 랜선을 꽂은 것처럼 연결하여 데이터 유출 경로를 차단함. **(망 분리 보안)**

**7. `06_07_file_share.tf`**
*   **역할:** VM들이 같이 쓰는 공유 폴더(NAS) 생성.
*   **핵심 기술:** **SMB Protocol & Shared Volume**
    *   여러 대의 웹 서버(VMSS)가 설정 파일이나 업로드 파일을 실시간으로 공유할 수 있게 해주는 공용 공간.

**8. `06_08_premium_storage.tf`**
*   **역할:** 고성능 전용 스토리지 별도 생성.
*   **핵심 기술:** **SSD-based Performance**
    *   속도가 느린 HDD(Standard) 대신 빠른 SSD(Premium)를 사용하여, I/O가 많은 작업에서의 병목 현상을 해결함.

### 📂 07. Database (데이터베이스 및 분석)

**9. `07_01_mysql_server_and_database.tf`**
*   **역할:** MySQL 메인 서버 생성.
*   **핵심 기술:** **Zone-Redundant HA (고가용성)**
    *   서버를 1대만 띄우는 게 아니라, 대기 서버(Standby)를 다른 존에 숨겨두고 장애 시 자동으로 교체(Failover)되게 만듦. **(서비스 연속성)**

**10. `07_02_read_replica.tf`**
*   **역할:** 읽기 전용 복제 DB 서버 생성.
*   **핵심 기술:** **Scale-Out (수평 확장)**
    *   조회(Select) 쿼리만 처리하는 서버를 따로 두어, 메인 서버가 쓰기(Insert/Update) 작업에만 집중하게 함. **(부하 분산)**

**11. `07_03_configuration.tf`**
*   **역할:** MySQL 내부 설정값(my.cnf) 튜닝.
*   **핵심 기술:** **Parameter Tuning**
    *   `slow_query_log` 등을 켜서 느린 쿼리를 잡아내고, 보안/성능 관련 커널 파라미터를 최적화함.

**12. `07_04_private_endpoint.tf`**
*   **역할:** DB를 내부망(VNet)에 연결.
*   **핵심 기술:** **Database Isolation (DB 격리)**
    *   DB 서버에 공인 IP를 아예 부여하지 않음. 해커가 인터넷을 통해 DB에 접속할 구멍 자체가 없음. **(최고 수준 보안)**

**13. `07_05_redis_cache.tf`**
*   **역할:** 메모리 기반 캐시 서버 생성.
*   **핵심 기술:** **In-Memory Computing**
    *   디스크(DB)에서 데이터를 꺼내는 것보다 수천 배 빠른 메모리(RAM)에서 데이터를 꺼내줌. **(초고속 응답)**

**14. `07_06_data_factory.tf`**
*   **역할:** 데이터 이동/변환 도구(ETL) 생성.
*   **핵심 기술:** **Serverless Data Integration**
    *   별도의 백업 서버를 구축하지 않고, 클라우드 관리형 서비스로 DB 데이터를 스토리지로 퍼 나르는 파이프라인 구축. **(운영 자동화)**

**15. `07_07_auditing_policy.tf`**
*   **역할:** 누가 DB에서 뭘 했는지 감시하는 정책.
*   **핵심 기술:** **Compliance & Auditing (감사 추적)**
    *   "누가 언제 테이블을 지웠나?" 같은 사고 발생 시 범인을 찾을 수 있는 로그 기록 시스템 가동.

**16. `07_08_cosmosdb.tf`**
*   **역할:** NoSQL 데이터베이스 생성.
*   **핵심 기술:** **Polyglot Persistence (폴리글랏 저장소)**
    *   정형 데이터(회원정보)는 MySQL에, 비정형 데이터(로그, 채팅 등)는 CosmosDB에 저장하여 데이터 성격에 맞는 최적의 저장소를 사용함. **(확장성 대비)**
